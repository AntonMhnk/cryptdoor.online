# План: Windows-версия + автоматическая сборка через GitHub

## 0. Что ты делаешь до начала (15-20 минут)

Эти вещи нужны **перед** тем как мы начнём кодить. Без них мы упрёмся.

### 0.1. Решить про GitHub-репозиторий

Сейчас `cryptdoor-app` лежит у тебя только локально. Нужно создать репу на GitHub.

**Варианты:**
- **Private** (по умолчанию рекомендую) — никто не видит код, релизы видны только тем кому дашь ссылку
- **Public** — код открытый, любой может посмотреть, релизы публичные

Имя репы — например `cryptdoor` или `cryptdoor-app`.

**Что сделать:**
1. Зайти на github.com → New repository
2. Создать пустую репу (без README, без .gitignore — мы добавим)
3. Запомнить URL вида `git@github.com:твой_логин/cryptdoor.git`

### 0.2. Решить про подписи (можно пропустить)

| Что | Стоит | Эффект | Решение сейчас |
|---|---|---|---|
| Apple Developer ID | $99/год | Mac не будет ругаться при первом запуске | **Пропускаем**, друзьям объяснишь команду `xattr -cr` |
| Windows EV cert | $300+/год | SmartScreen не будет ругаться | **Пропускаем**, друзья кликнут "Run anyway" |

Если потом захочешь — добавим в любой момент, не критично.

### 0.3. Где будем тестить .exe

**Реальные варианты в порядке удобства:**
1. **Друг с Windows** — самое простое, ему скинул .exe → он поставил, написал что работает/не работает
2. **UTM на Mac** (бесплатно) — поставить Windows 11 ARM в виртуалке, тестить локально. Минус — медленно на M-чипах из-за эмуляции x64
3. **Parallels Desktop** ($100/год) — самый шустрый вариант VM на Mac
4. **Реальный Windows-комп** если у тебя есть

Минимум нужен **один** способ протестировать перед раздачей друзьям.

### 0.4. Решить как раздавать ссылку друзьям

GitHub Releases будут давать прямую ссылку вида:
```
https://github.com/твой_логин/cryptdoor/releases/latest
```

Здесь будут оба файла:
- `CryptDoor_0.1.0_aarch64.dmg` — для Mac на Apple Silicon
- `CryptDoor_0.1.0_x64-setup.exe` — для Windows

Друзья заходят, видят свою систему, качают.

Если хочешь чтобы было ещё проще — можно сделать страничку на твоём домене `cryptdoor.online/download` которая редиректит на нужный файл по типу системы. Но это уже на потом.

## 1. Что я делаю (по фазам)

### Фаза 1 — Подготовка проекта к Git (30 минут)

- Создать `.gitignore` (node_modules, target, dist, sidecar бинарники качаются скриптом)
- Создать нормальный `README.md`: что это, как ставить, как пользоваться, скриншоты
- `git init` + первый коммит
- `git remote add origin <твоя URL>`
- `git push -u origin main`

После этого код будет на GitHub. Всё что было локально — там.

### Фаза 2 — Windows-код (4-6 часов работы)

Один проект, один код, разные системы. Rust сам выберет нужное через `#[cfg(target_os = "...")]`.

**Что меняется:**

1. **Helper IPC** (`src-tauri/src/core/helper_client.rs`)
   - Сейчас: Unix domain socket `/var/run/online.cryptdoor.helper.sock`
   - Под Windows добавляется: Named Pipe `\\.\pipe\online.cryptdoor.helper`
   - Один и тот же протокол сообщений (JSON), разный транспорт

2. **Helper daemon** (`src-tauri/src/bin/helper.rs`)
   - Сейчас: standalone бинарник, запускается через launchd
   - Под Windows: тот же бинарник запускается как **Windows Service** (постоянно крутится в фоне как сервис системы)
   - Используем крейт `windows-service` для регистрации/удаления сервиса

3. **Установка helper** (`src-tauri/src/commands.rs`)
   - Сейчас на Mac: `osascript` запрашивает админ-пароль, копирует helper в `/Library/PrivilegedHelperTools/`, регистрирует в launchd
   - Под Windows: запускаем `helper.exe install` с флагом UAC (`runas`), он сам себя регистрирует как Service и стартует

4. **mihomo + wintun** (`scripts/download-mihomo.mjs`)
   - Сейчас качает Mac-бинарник с GitHub releases
   - Добавим: качает `mihomo-windows-amd64.exe` + `wintun.dll` (TUN-драйвер от Wireguard)
   - `wintun.dll` обязательно должен лежать рядом с `mihomo.exe` — иначе TUN не работает на Windows

5. **Tauri bundle** (`src-tauri/tauri.conf.json`)
   - Добавить `nsis` в `targets` — это формат Windows-установщика
   - Прописать ресурсы: `mihomo.exe`, `wintun.dll`, `cryptdoor-helper.exe`
   - NSIS-хук: при установке регистрирует helper как сервис, при удалении — снимает

6. **Иконки**
   - У нас уже есть `icon.ico` в проекте — используется для Windows автоматически
   - Tray-иконка та же (PNG работает)

7. **Tauri-плагин autostart**
   - Уже работает на обеих системах. На Windows пишет ключ в реестре `HKCU\...\Run`. Ничего не меняем.

### Фаза 3 — GitHub Actions CI (2 часа)

Файл `.github/workflows/release.yml` — запускается когда пушим тэг `v0.1.0`:

```yaml
on:
  push:
    tags: ['v*']

jobs:
  build:
    strategy:
      matrix:
        platform: [macos-latest, windows-latest]
    runs-on: ${{ matrix.platform }}
    steps:
      - checkout
      - setup rust + node
      - pnpm install
      - download mihomo (для нужной OS)
      - pnpm tauri:build
      - upload .dmg / .exe в GitHub Release
```

После этого процесс будет такой:

```bash
# поправил что-то в коде
git commit -am "fix: что-то"
git push

# готов релизить новую версию
git tag v0.1.1
git push --tags
# → через 10-15 минут на GitHub Releases лежат свежие .dmg + .exe
```

В Releases автоматически генерится changelog по коммитам.

### Фаза 4 — Тестирование (1-2 часа)

- Скачиваем `.exe` из Release
- Ставим в VM или у тебя на втором компе или у друга
- Проверяем:
  - Установщик не падает
  - SmartScreen ругается, но "Run anyway" работает
  - Иконка в трее появляется
  - При первом Connect выскакивает UAC — соглашаемся
  - IP меняется → проверяем `whoer.net`
  - Перезагрузили Windows → CryptDoor сам стартанул в трее
  - Telegram работает

Если что-то не так — фикс, новый тэг, новый релиз.
